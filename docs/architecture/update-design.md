## 更新与版本管理设计（Desktop 与封装 OpenClaw）

本设计文档聚焦：「用户安装 OpenClaw Desktop 之后，如何**手动/自动更新 Desktop 本身**，以及**随应用捆绑的 OpenClaw 如何更新**」。

---

### 1. 目标与原则

- **统一体验**：用户只需要「更新 Desktop」，就能在绝大多数情况下同时获得较新的 OpenClaw 能力。
- **可控风险**：OpenClaw 升级可能带来不兼容变更，需要有明确的版本策略和回滚/锁定机制。
- **区分责任边界**：
  - Desktop 负责打包、分发、检测更新、触发更新；
  - OpenClaw 核心项目负责语义版本与兼容性承诺。

---

### 2. 国内网络环境问题与设计约束

依赖 GitHub Releases 作为唯一更新源时，在国内会面临两类主要问题：

| 问题 | 表现 | 影响 |
|------|------|------|
| **访问不稳定** | 无法连接 api.github.com、连接超时、间歇性失败 | 检查更新失败、用户误以为「没有更新」或频繁报错 |
| **下载速度慢** | GitHub 资源服务器在海外，直连带宽有限 | 更新包下载极慢或中断，自动更新体验差 |

因此，更新机制设计需要**显式考虑国内用户**，在文档与实现上支持「可替代更新源」与「不依赖 GitHub 的更新路径」。

---

### 3. Desktop 更新机制设计

#### 3.1 自动更新（推荐路径）

- **基础方案**
  - 使用 `electron-updater` 连接 GitHub Releases：
    - 启动 Desktop 时检查当前版本与最新可用版本；
    - 如有新版本，可配置为：
      - 弹出提示框：展示版本号与简要 Changelog，用户选择「立即更新」或「稍后提醒」；
      - 或静默下载更新，在下次重启时完成更新（适合团队内部分发）。

- **信息流**
  - GitHub Releases 中以 `vX.Y.Z` 形式发布版本；
  - 每个 Release 包含：
    - 各平台安装包 / 自动更新所需差分包；
    - Release Note：说明变更内容、包含/依赖的 OpenClaw 版本等。

- **UI 展现**
  - 在 Desktop 中：
    - 顶部/底部状态栏显示「有可用更新 vX.Y.Z」；
    - 点击进入「更新详情弹窗」，展示：
      - 新版本号；
      - 核心变更列表；
      - 封装的 OpenClaw 版本区间（例如：`OpenClaw >= 0.9, < 1.0`）。

#### 3.2 手动更新（备选路径）

- **从 GitHub Releases 手动下载**
  - 用户主动打开 GitHub 仓库 Releases 页面；
  - 下载对应平台安装包（.dmg / .exe / AppImage 等）；
  - 按操作系统常规方式安装/替换已有应用。

- **在应用内提供「检查更新」入口**
  - 在 `Settings` 或 `About` 页面提供按钮：
    - 点击后立即触发一次更新检查；
    - 显示当前版本、最新版本、更新说明；
    - 提供「前往下载页面」链接（适用于无法直接自动更新的环境）。

---

### 4. 国内 / 多源更新方案（应对 GitHub 不可用与慢速）

在保留 GitHub Releases 作为默认源的前提下，通过「可配置更新源 + 多源回退 + 手动下载多链接」提升国内可用性与下载速度。

#### 4.1 方案概览

| 方案 | 说明 | 适用场景 | 实现复杂度 |
|------|------|----------|------------|
| **A. 可配置更新源** | 通过配置（环境变量 / 应用设置）指定更新服务器，使用 electron-updater 的 **generic** provider | 自建/镜像站、企业内网 | 中 |
| **B. 多源回退** | 先请求国内镜像或自建 API，失败再回退 GitHub | 国内用户为主、希望自动更新仍可用 | 中高 |
| **C. 手动下载多链接** | 应用内「检查更新」仅提示有新版本 + 提供**多个下载入口**（GitHub + 国内镜像/网盘） | 快速落地、不改 updater 核心逻辑 | 低 |

建议：**短期采用 C + 文档说明**；**中期实现 A 或 B**，使国内用户可指向镜像或自建服务。

#### 4.2 方案 A：可配置更新源（generic provider）

- **思路**
  - electron-updater 除 GitHub 外支持 [generic provider](https://www.electron.build/auto-update#custom-server)：向指定 URL 请求 `latest.yml`（或平台相关 yml）及安装包文件。
  - 在 Desktop 中增加「更新源」配置项（或读取环境变量），例如：
    - `default`：使用 GitHub（当前行为）；
    - `custom`：使用配置的 base URL（如 `https://releases.example.com/openclaw-desktop/`）。
  - 构建/分发时：为「国内版」或企业版打包可预填镜像 base URL 的配置，用户无需改设置即可走镜像。

- **镜像端需要提供**
  - 与 electron-builder 产出一致的目录结构，例如：
    - `latest-mac.yml` / `latest.yml`（含版本号、文件名、sha512 等）；
    - 各平台安装包：`.dmg`、`.exe`、`.AppImage` 等。
  - 可通过 CI 在发布到 GitHub 后，同步上传到自建对象存储（如阿里云 OSS、腾讯云 COS）+ CDN，或使用 Gitee Releases 等。

- **配置示例（设计）**
  - 环境变量：`OPENCLAW_UPDATE_BASE_URL=https://releases.example.com/openclaw-desktop/`
  - 或应用内设置：`settings.update.baseUrl`，存于用户配置文件中。

#### 4.3 方案 B：多源回退

- **思路**
  - 检查更新时，按顺序请求多个源（例如先国内镜像 API，再 GitHub）；
  - 任一源返回「有新版本」即展示更新提示；
  - 下载时使用该源提供的安装包 URL（若镜像提供则从镜像下载，避免走 GitHub 直连）。
  - 需要维护一份「当前版本与各源最新版本」的兼容逻辑，避免多源版本不一致导致误判。

- **实现要点**
  - 更新检查：请求镜像的 `latest.yml` 或简单 version API，失败则请求 GitHub Releases API；
  - 下载：若选择镜像源，则从镜像 URL 下载；否则走 GitHub。
  - 可在 UI 上标注「当前更新源」或「因网络原因使用备用源」，便于用户理解。

#### 4.4 方案 C：手动下载多链接（立即可做）

- **不改动自动更新逻辑**，仅增强「手动更新」体验与文档：
  - 在 **设置 / 关于** 中「检查更新」：
    - 若检测到新版本（在能连上 GitHub 时），弹窗中除「立即更新」外，增加「**手动下载**」按钮；
    - 点击后打开**帮助页或文档页**，页面上列出：
      - GitHub Releases 链接（海外用户）；
      - **国内可访问的下载链接**（见下）。
  - 若**无法连接 GitHub**（检查更新失败）：
    - 在 UI 中提示「当前无法检查更新，您可前往下方链接查看是否有新版本」；
    - 同样提供「打开下载页」入口，下载页提供多链接。

- **国内下载链接可选来源**
  - **Gitee 镜像仓库**：在 Gitee 建镜像仓库，Release 与 GitHub 同步（需 CI 或手动同步安装包）；
  - **对象存储 + CDN**：将每次 Release 的安装包同步到阿里云 OSS / 腾讯云 COS 等，配置 CDN，在文档中提供固定页面链接；
  - **网盘**：将稳定版安装包上传至蓝奏云、阿里云盘分享等，在文档中注明「国内用户若 GitHub 慢可从此处下载」（需人工维护）。

- **文档与 FAQ**
- 在 `docs/support/faq-and-troubleshooting.md` 中增加「国内用户如何更新」：
    - 说明 GitHub 可能不稳定、下载慢；
    - 给出「手动下载」步骤及上述多链接入口；
    - 若有镜像或 Gitee，明确写出链接。

#### 4.5 推荐实施顺序

1. **近期**：落实方案 C（多链接 + 文档 + 检查更新失败时的友好提示），不依赖国内网络即可改善体验。
2. **中期**：若存在稳定镜像或自建 Release 服务，实现方案 A（可配置 base URL + generic provider），为「国内版」或企业版预设镜像 URL。
3. **可选**：在方案 A 成熟后，再考虑方案 B（多源自动回退），进一步减少用户感知到的失败率。

### 5. 封装 OpenClaw 的更新策略

#### 5.1 封装架构回顾

- Desktop 中通过 `bundle:node` 等脚本，将：
  - Node.js 运行时；
  - NPM `openclaw` 包；
  - 及其依赖；
  - 一同放入 `resources/node/`（构建后迁移到应用资源目录）。
- 内置终端与后台 Gateway 进程使用这份 Node + OpenClaw 环境。

#### 5.2 版本绑定策略选型

设计两个层面的策略：

1. **Desktop ↔ OpenClaw 版本绑定策略**
   - **方案 A：强绑定（推荐初期阶段）**
     - 每个 Desktop 版本都内置一个经过测试的 OpenClaw 版本；
     - Desktop 的 SemVer 次版本升级（`MINOR`）至少保证 OpenClaw 版本不会回退；
     - Release Note 中明确写明当前内置的 OpenClaw 版本：
       - 例如：`Embedded OpenClaw: 0.9.3`。
   - **方案 B：弱绑定 + 动态更新（可作为后续增强）**
     - Desktop 启动时检查 OpenClaw 包是否有新版本；
     - 允许在保持 Desktop 版本不变的情况下，仅更新 OpenClaw；
     - 需要引入更复杂的兼容性检查和回滚机制（不建议一开始就实施）。

2. **OpenClaw 包更新策略**
   - 初期建议：
     - 仅通过 Desktop 的整体升级来更新内置 OpenClaw；
     - 避免在用户机器上静默地 `npm install openclaw@latest` 这类不可控行为；
     - 减少「Desktop 与 OpenClaw 版本不匹配」导致的隐性问题。

#### 5.3 用户视角的「OpenClaw 版本」信息呈现

- 在 Config 或 About 页面显示：
  - Desktop 版本：`OpenClaw Desktop vX.Y.Z`；
  - 内置 OpenClaw 版本：`OpenClaw Core vA.B.C`；
  - Node.js 运行时版本：`Node vN.M.P`（可选）。
- 在自动更新弹窗中，简要提示：
  - 本次更新是否包含 OpenClaw 升级；
  - 若包含，是否有兼容性注意事项（例如配置格式变更）。

### 6. 典型更新流程（用户旅程）

#### 6.1 自动更新场景

1. 用户启动 Desktop；
2. Desktop 按配置检查更新（默认 GitHub；若配置了镜像则可能先请求镜像，参见 §4）；
3. 若发现新版本：
   - 在 UI 中提示「发现新版本 vX.Y.Z（包含 OpenClaw vA.B.C）」；
   - 用户点击「立即更新」；
4. Desktop 下载更新包（差分或完整包）；
5. 下载完成后提示「重启以应用更新」；
6. 重启后，Desktop 与内置 OpenClaw 一起升级到新版本。

#### 6.2 手动更新场景

1. 用户在应用内点击「检查更新」；
2. Desktop 调用更新 API 返回「有新版本」；
3. 用户点击「前往下载页面」，打开 GitHub Releases；
4. 用户下载并安装新的安装包；
5. 启动后，Desktop 与内置 OpenClaw 同步升级。

---

### 7. 回滚与兼容性设计（预留）

#### 7.1 回滚策略（未来可选）

- 若新版本 Desktop+OpenClaw 导致严重问题，可考虑：
  - 在 Releases 页面保留最近若干稳定版本供下载；
  - 在文档中指引用户下载旧版本安装包进行「手动回滚」。
- 高级方案（需要额外开发）：
  - 本地保留上一个已知可用版本；
  - UI 中提供「回退到上一个版本」的选项；
  - 但这会大幅增加实现复杂度与测试成本，初期可以不做。

#### 7.2 配置兼容性

- OpenClaw 的配置（`~/.openclaw/openclaw.json`）可能随版本演进而变化：
  - Desktop 升级时需要注意配置迁移与兼容提示；
  - 若检测到配置版本与当前 OpenClaw 版本不匹配，可以：
    - 提示用户进行备份与迁移；
    - 提供最小化的自动迁移逻辑或工具（由 OpenClaw 核心项目主导）。

---

### 8. 实现建议与任务拆分

- **短期（v0.2.x）**
  - 明确 Desktop 与内置 OpenClaw 的版本绑定策略（采用强绑定）；
  - 完善自动更新提示与 Release Note 中的版本说明；
  - 在 UI 中展示当前 Desktop 版本与内置 OpenClaw 版本；
  - **国内网络**：落实 §4.4 方案 C（检查更新失败时友好提示 + 「手动下载」多链接入口 + FAQ「国内用户如何更新」）。

- **中期（v0.3.x+）**
  - **国内网络**：若具备镜像或自建 Release 服务，实现 §4.2 可配置更新源（generic provider），或 §4.3 多源回退；
  - 评估是否需要支持「在 Desktop 不升级的情况下，单独升级 OpenClaw」；
  - 若需要，引入：
    - OpenClaw 版本兼容性矩阵；
    - 安全可控的「在线更新 OpenClaw 包」机制；
    - 回滚能力。
